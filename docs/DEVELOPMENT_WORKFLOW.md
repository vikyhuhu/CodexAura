# CodexAura 开发与发布流程

这套流程的目标不是要求项目所有者审核代码，而是把职责分开：CI 和 Codex 负责技术检查，项目所有者负责确认产品结果。

## 四个核心概念

| 名称 | 在本项目中的含义 |
|---|---|
| `main` | 随时应当可以发布的稳定版本 |
| 功能分支 | Codex 实现单个需求的独立工作区 |
| PR | 等待检查和验收的候选版本 |
| CI | GitHub 自动执行的测试、构建和安全门禁 |

## 标准流程

1. 项目所有者用自然语言说明需求和验收标准。
2. Codex 创建 `agent/<说明>` 分支，不直接修改 `main`。
3. Codex 实现功能、补充测试，并在本地完成验证。
4. Codex推送分支并创建 Draft PR，用中文说明改动、影响、风险和回滚方式。
5. GitHub CI 自动运行单元测试、双架构 App 构建、签名与资源检查。
6. CI 全绿后，项目所有者只需按照 PR 中的验收步骤检查产品结果。
7. 项目所有者明确回复“可以合并”后，Codex 才合并 PR。
8. 合并后再次确认 `main` CI 通过；若出现问题，回滚该次合并。

## CI 检查内容

- `unit-tests`：运行 `./Scripts/test.sh`。
- `build-app`：运行 `./Scripts/build-app.sh`，并检查 arm64/x86_64、代码签名和五套内置预设。
- `quality-gate`：汇总前两项；任意一项失败时，它也会失败。

`main` 的保护规则要求 `quality-gate` 通过，避免未经自动验证的代码直接进入稳定分支。

## 项目所有者如何验收

不需要阅读代码，只需要检查 PR 的“给产品验收”部分：

- 用户看到的界面和行为是否符合需求。
- 提供的操作步骤能否正常完成。
- 文案、名称和业务规则是否正确。
- 风险和回滚方法是否已经解释清楚。

如果看不懂说明，直接要求 Codex 改写成非技术语言；不要在没有理解产品影响时确认合并。

## 风险分级

- 低风险：文案、颜色、静态资源。CI 通过并完成视觉验收即可。
- 中风险：主题导入、配置兼容、UI 行为。需要自动测试和清晰的手动验收步骤。
- 高风险：签名校验、CDP 安全边界、文件删除、凭证和发布流程。需要失败场景测试、明确回滚方案和单独确认。

## CI 失败时

不要绕过门禁，也不要直接向 `main` 推送。让 Codex读取失败日志、解释原因、在原分支修复并重新运行 CI。只有 `quality-gate` 变绿且产品验收完成后才能合并。
